# LogueOS-Companion — Audit Findings (2026-06-02)

Evidence-based audit: 7 parallel agents over the real code (every finding cited to file:line) + a lead-architect synthesis. Test suite re-run during the audit: **134/134 pass, 1 type error**.

## Headline

Sully is in genuinely good structural health for a fast-moving solo-operator fork — the team is actively winning the fight against legacy duplication (shared chat_turn/stream_prepare/chat_prompt services, a clean run-mode config, well-decomposed Svelte controllers, 134 green tests). But three things undercut that: (1) a real latent runtime bug where the chat_activity database index is built with different columns depending on boot order; (2) the original /api/chat handler is stuck half-migrated and is the source of the two scariest behavior bugs — it logs replies to the wrong place AND can feed a dispatched worker the WRONG conversation's history; and (3) the safety net is unplugged: 134 passing tests, a type-checker, and lint all exist but NOTHING runs them in CI, so a broken build can sail straight to your iPhone. None of this is on fire today, but the gaps are exactly the kind that bite silently. The autonomous-dispatch 'teacher' design also doesn't work the way its own comments claim — Sully's own judgment to escalate is currently vetoed by a brittle keyword regex.

## Refactor themes

### Unplug-proof the safety net (CI + the one type error blocking it)
You already paid for the quality gates — 134 tests pass, there's a type-checker, there's lint — but none of them run automatically. A push that breaks types or fails tests reaches TestFlight with the iOS build as the only gate. This is the single highest-leverage, lowest-risk area: turn existing-but-unenforced work into an actual guardrail. One real type error (a Buffer handed to a web Response in the voice/speak path) must be fixed first, or the gate is red on day one.

### Fix the silent data-layer bug and tame DB connection sprawl
chat_activity builds an index whose columns depend on which module touches the cold database first — same index name, different columns in two files. That's nondeterministic at runtime, not a style nit. Underneath it, ~23 modules each open and close their own database connection per call (a single chat turn opens the DB a dozen-plus times) with no shared busy_timeout, so concurrent writers can collide. The index drift is a 5-minute fix; the connection consolidation is the bigger structural cleanup that also kills several duplicated table definitions at once.

### Finish migrating the legacy /api/chat handler — it's the source of the real behavior bugs
The original 530-line /api/chat handler is stuck mid-migration toward the modern streaming architecture. Because it never got the shared assistant-turn helper, replies through it skip forensics, skip the task journal, and never refresh the thread's working-memory summary — silently degrading long-thread context. Worse, when you @cc-dispatch a worker from any non-default thread, the handler reads history from the WRONG thread (a missing argument on one line), so the worker can get a different conversation's context. These are the two highest-severity behavior bugs in the whole audit, and they live entirely in this one un-migrated file.

### Make Sully's autonomous-dispatch judgment actually work the way it's documented
The design intent is that Sully (on the Opus/Sonnet path) reads the conversation and decides whether to escalate to a coding worker. In reality her decision is ANDed with a brittle keyword regex: if she decides yes but your phrasing lacks an explicit file path or imperative+repo pair ('can you get the build green again?'), the dispatch is silently dropped. So the teacher can only agree or downgrade, never override — contradicting the code's own comments. Separately, a documented injection guard against auto-dispatching pasted/tool content is wired but permanently disabled (a flag hardcoded to false). And two state-machine transition bugs spam your logs and corrupt the forensic journal on exactly the failed tasks you'd most want to inspect.

### Consolidate the two voice stacks and make usage accounting honest
There are two near-complete voice subsystems (immersive realtime + legacy in-composer Talkback) with overlapping TTS-playback, mic, and blob-lifecycle code duplicated across both — a 'fixed the leak in one but not the other' hazard that has already bitten once per controller. The realtime path is now your primary workflow. Meanwhile the usage/cap counters only count cloud usage: local TTS and the realtime STT path record nothing, so your usage UI under-reports and a cap counter can freeze the moment it's hit. The realtime voice path itself (fall-forward, GPU lifecycle, CUDA-poison recovery, barge-in) is solid — the debt is duplication, dead code, and misleading numbers.

### iOS / push polish and frontend a11y quick wins
The build/push pipeline is robust and well-documented, but its fragility is concentrated in two script anchors that could break a previously-green build on a routine rebuild (because no native project is committed to diff against), plus operational blind spots: push failures are counted then thrown away, and a JWT cache can serve a stale key for ~50 min after a key rotation. On the frontend, the primary send key uses a deprecated event that misfires on some mobile keyboards — a real shippable behavior risk — plus a few additive accessibility gaps.

## Do NOT touch (load-bearing)

- The run-mode config matrix in config.ts (runMode booleans + appIdentity) — this is the cleanest part of the codebase, deliberately centralized, fail-closed default-wired, and unit-tested. Every kernel-coupled site reads one named boolean. Leave the pattern intact.
- The shared chat_turn.ts / stream_prepare.ts / chat_prompt.ts extractions — these exist specifically to stop the legacy+streaming paths drifting; both open with comments explaining why. The fix is to migrate MORE onto them, never to fork them again.
- The hot-window assembly in stream_prepare.ts (DESC+reverse, ORDER BY id not timestamp) — load-bearing fix for the model-amnesia / turn-7-empty-reply regression, with a CRITICAL comment forbidding regression. Don't 'simplify' the ordering.
- The dispatchStreams registry deliberately NOT being $state (+page.svelte:122-128) and the untrack() wrap on the stream-mirror $effect (streaming.svelte.ts:106) — these are documented Svelte 5 runes-trap workarounds; removing them reintroduces render-loop bugs.
- The pollMessages stale-response guard (captures requestedThread, drops stale in-flight responses after a thread switch) — looks like defensive over-engineering but prevents a real cross-thread message bleed.
- wav_pad.ts and tts_normalize.ts ordering — both are defensively correct (wav_pad never throws/corrupts on parse failure; tts_normalize's rule ordering is deliberate). The ~700ms WAV silence pad is an iOS clip-cutoff workaround, not dead code.
- The stable-signing-key injection in tools/trigger-ios-build.sh (key lives OUTSIDE git, injected per-build) — this is the hard-won fix for Apple's ~2-3 distribution-cert cap. Do not move it into the repo or persist it via the Codemagic API.
- The regenerate-ios-every-build model (ios/ and www/ gitignored, native project rebuilt via cap add + patch scripts) — intentional and documented. The patch-script anchors are fragile (worth hardening) but the overall approach is sound; don't 'commit the native project' as a fix.
- The tailnet-gated fail-closed auth on apns-register and other routes (rejects any Tailscale-Funnel-Request header) — this is the security boundary; the absence of app-level auth is correct-by-design for this deployment.
- The brakes chain ordering in dispatchToWorker (kill -> breaker -> daily cap -> fingerprint -> token bucket) and its exclusion of pre-dispatch task states from the daily-cap count — deliberately reasoned (stops task-first row-minting from falsely tripping the cap).

## Risks / caveats

- The legacy /api/chat handler is the single riskiest file to change — it carries 5 mutually-exclusive branches behind flag soup, so any edit can touch an unrelated branch. Migrate it incrementally (one branch at a time) behind the new CI gate, not in one big rewrite.
- Making Sully's gate-block authoritative (theme 4 / step 12) CHANGES her autonomous behavior — she'll start dispatching workers on phrasing that's currently vetoed. This is a product decision, not just a refactor; confirm desired behavior with the operator before shipping, or you may get unexpected worker spawns (each burning Max quota).
- The shared-db.ts refactor (step 14) touches ~23 modules and the boot path. It's the right call but it's an L-effort change to the data layer — do it AFTER CI exists so the 134 tests catch regressions, and verify WAL + busy_timeout behavior, not just that it compiles.
- Several findings are explicitly scaled for a single-operator, single-device deployment (device_id 'ios-app', serial APNs fan-out, cap TOCTOU races). They are low-priority now but would escalate immediately if Sully ever ships to a second device — flag that boundary before any multi-device plan.
- Several 'deferred by design' items (proposed->synthesized arc never closed for pure-chat turns, the synthesis journal vocabulary built ahead of its driver) assume a future Phase 3. Before any consumer is built that assumes a closed task arc, verify Phase 3 is actually scheduled — the FSM is built well ahead of the code that drives it.
- The audits were static reads (plus the test suite + svelte-check, which I re-ran and confirmed: 134/134 pass, 1 type error). The Python speech services, the WS faster-whisper STT server, and the dispatch listener itself were out of scope and not executed — claims about those edges are inferred from client wiring, so validate against running services before acting on voice/dispatch-transport changes.
- Browser-load QA on the real iPhone viewport is mandatory after ANY frontend change here (onkeydown fix, a11y, WorkingBubble) — endpoints returning 200 and tests passing have previously masked operator-visible UI regressions in this codebase.

## Full findings by dimension

### Server Architecture & Wiring (src/lib/server/* + src/routes/api/chat/*)

**Wiring:** Boot is triggered from src/hooks.server.ts (the only server entry point): at module load it calls bootstrapCompanionDb() (creates the two kernel-made tables + a handful of Phase-1 tables, runs additive ALTER migrations) then conditionally startCompletionPoller() — both gated by runMode flags from config.ts. config.ts is the single source of truth for run-mode coupling: it reads env via a $env/dynamic/private → process.env → default chain and derives a runMode boolean matrix ('wired' → all kernel features on, 'companion' → off, fail-closed default-wired) plus an appIdentity object (fork-aware labels/basePath/defaultWorkspace). Every kernel-coupled call site reads ONE named boolean from runMode rather than string-comparing the mode — this is clean.

The chat pipeline has TWO live entry points that do overlapping work: (1) the LEGACY src/routes/api/chat/+server.ts POST — a ~530-line god-handler with 5 mutually-exclusive branches (companion-dispatch, kernel-gateway dispatch, Hermes, conversational llm_router, image-gen) selected by a tangle of boolean flags (shouldTrigger/shouldDispatch/shouldRouteChat/isHermes/isAgyChat) and the gateway worker-prompt is a 75-line template literal inline in the handler; and (2) the MODERN src/routes/api/chat/sdk-stream/+server.ts — a thin orchestrator that delegates its preamble to chat/stream_prepare.ts (persist → classify → hot-window assembly → provider/model resolve → CLI-vs-direct decision → tool gating → system prompt) and its post-reply dispatch to chat/autonomous_dispatch.ts. A THIRD partial pipeline lives in api/chat/voice-reply/+server.ts which reuses the shared chat_turn services + detectTargetRepo but re-implements its own hot-window. Shared services were extracted well: chat_turn.ts (persistUserTurn/classifyAndTouchThread/persistAssistantTurn/mintTaskId), chat_prompt.ts (buildSystemPrompt — single copy, both paths call it), model_catalog.ts (resolveChatModel), config.ts. Data access is the weak spot: ~23 modules each open their own better-sqlite3 connection to the SAME logueos_memory.db file via a per-call `new Database()`/close pattern, and table-schema ownership is scattered across ~15 modules (each owns its own CREATE TABLE IF NOT EXISTS + lazy ensure-guard).

**Strengths:**
- ✅ Run-mode coupling is genuinely well-designed: config.ts:182-195 derives a single runMode boolean matrix from the mode string, every kernel-coupled site reads ONE named boolean (runMode.dispatchEnabled, .completionPoller, etc.) instead of scattering `mode === 'companion'` compares, fail-closed default-wired, and a comment notes a unit test asserts the full matrix. appIdentity (config.ts:203-224) similarly centralizes fork-aware identity so no route hard-codes 'LogueOS-Console'.
- ✅ Shared-service extraction against drift is deliberate and documented: chat_turn.ts and chat_prompt.ts both open with a comment explaining they exist because the legacy + sdk-stream routes used to keep lockstep copies that drifted. buildSystemPrompt is now a single source both paths call (chat_prompt.ts:4-5).
- ✅ stream_prepare.ts is a clean preamble extraction — the route handler (sdk-stream/+server.ts) stays a thin parse→prepare→branch orchestrator, and the load-bearing hot-window assembly (the model-amnesia fix) is isolated in one place with a CRITICAL comment forbidding regression (stream_prepare.ts:13-16, 126-153).
- ✅ Server-only config isolation is correct: config.ts lives under $lib/server/ with a header explaining the split from a client-importable $lib/config.ts, and clientSafeConfig (config.ts:228-249) explicitly whitelists only non-path fields for the client load().

**Issues:**
- 🔴 **[high]** Triplicated/drifting table schemas: pending_jobs and chat_activity are each defined in two modules, and the two chat_activity index definitions disagree
  - pending_jobs full schema + additive-migration loop is defined TWICE — bootstrap.ts:141-184 and dispatchJobs.ts:94-133 — and both must be hand-kept in sync (bootstrap even comments 'dispatchJobs.ts's getDb() also creates it'). Worse, chat_activity is defined in bootstrap.ts:87-94 AND chatActivity.ts:8-17, and the two create the SAME-NAMED index with DIFFERENT columns: bootstrap.ts:94 is `idx_chat_activity_trace ON chat_activity(trace_id)` while chatActivity.ts:16 is `idx_chat_activity_trace ON chat_activity(trace_id, timestamp)`. Whichever module touches the cold DB first wins; CREATE INDEX IF NOT EXISTS silently no-ops the second, so the index that actually exists depends on boot ordering. This is a real latent bug, not a style nit — the columns covered by that index are nondeterministic.
  - _evidence:_ `src/lib/server/bootstrap.ts:87-94,141-184; src/lib/server/dispatchJobs.ts:94-133; src/lib/server/chatActivity.ts:8-17`
- 🟡 **[med]** ~23 modules each open their own per-call SQLite connection to the same DB file; no shared connection or pooling
  - chat.ts:6-8, thread_state.ts:64-66, dispatchJobs.ts:91-92, chatActivity.ts:58, thread_meta.ts, usage.ts, web_push.ts, apns.ts, semantic.ts, memory.ts, settings.ts, observation_emit.ts, workspace_context.ts, chat_drafts.ts, dispatchBrakes.ts, dispatchUsage.ts, voice_usage.ts, web_usage.ts, turn_replay.ts, plus 4 route files — all do `new Database(serverConfig.memoryDbPath)` per function call then close in a finally. A single chat turn opens/closes the DB a dozen-plus times (persistUserTurn, classifyAndTouchThread, getChatMessages x2, thread_meta, thread_state, chatActivity per journal event, dispatchJobs). WAL is set ONCE in bootstrap.ts:38 but the per-call openers do NOT re-assert journal_mode/busy_timeout, so concurrent writers (the worker activity-callback POST hitting chatActivity while a stream persists a reply) can hit SQLITE_BUSY with no configured busy_timeout. There is no shared module owning the connection. This is the single biggest structural fragility in the data layer.
  - _evidence:_ `src/lib/server/chat.ts:6-8; src/lib/server/thread_state.ts:64-66; src/lib/server/dispatchJobs.ts:91-92; src/lib/server/bootstrap.ts:34-38 (WAL set only here)`
- 🟡 **[med]** Legacy /api/chat POST is a god-handler: 5 branches, flag soup, and a 75-line inline worker-prompt
  - src/routes/api/chat/+server.ts POST (lines 72-600) carries the entire legacy dispatch surface in one function: it resolves worker + role + targetRepo by ad-hoc substring matching (123-152), then computes shouldDispatch/shouldRouteChat/shouldTrigger/isHermes/isAgyChat/hasExplicitDispatchIntent (178-201) and runs five non-trivial branches (companion-native dispatch 207-243, companion-no-dispatch notice 246-257, Hermes 263-301, conversational llm_router 310-375, image-gen 380-397, then the kernel-gateway dispatch 399-583). The worker prompt is a 75-line template literal inline at 422-499. Unlike the sdk-stream path it does NOT use prepareStream/stream_prepare, so its repo-detection (140-152) and NEW-CONVERSATION slicing (313-326, 405-417) are hand-duplicated. Hard to test, hard to change without touching unrelated branches.
  - _evidence:_ `src/routes/api/chat/+server.ts:72-600 (branch flags 178-201; inline worker prompt 422-499)`
- 🟡 **[med]** Two divergent dispatch transports with no unifying abstraction
  - There are two completely separate ways a worker gets spawned. The legacy route dispatches via HTTP POST to the kernel gateway `${serverConfig.gatewayUrl}/api/v1/dispatch` with a standard_worker tool_profile (+server.ts:506-527), while the companion path dispatches via dispatchToWorker() → companionDispatch.ts → the dispatch listener with HMAC (autonomous_dispatch.ts:80-88, +server.ts:212-220). They are gated by different runMode flags (dispatchEnabled vs companionDispatchEnabled), build the worker brief differently, and write different system-message copy. No shared 'dispatch a worker' interface — the two will drift on auth, retry, and error-surfacing semantics.
  - _evidence:_ `src/routes/api/chat/+server.ts:506-527 (gateway path); src/lib/server/chat/autonomous_dispatch.ts:79-110 + src/lib/server/companionDispatch.ts (listener path)`
- ⚪ **[low]** Repo-detection heuristic and NEW-CONVERSATION slicing duplicated across three pipelines
  - detectTargetRepo() is exported from stream_prepare.ts:33-47 and reused by sdk-stream and voice-reply, but the LEGACY route open-codes the identical miru/orchestrator/nasdoom/console keyword scan inline at +server.ts:140-152 instead of importing it. Likewise the 'slice history after the last --- NEW CONVERSATION --- marker' logic appears at least three times: +server.ts:269-285 (Hermes), 313-326 (chat), 405-417 (dispatch), each a slightly different hand-rolled copy. The hot-window assembly also exists in stream_prepare.ts:126-153 (HOT_WINDOW=20) and is re-implemented in voice-reply with its own HISTORY constant.
  - _evidence:_ `src/lib/server/chat/stream_prepare.ts:33-47,126-153; src/routes/api/chat/+server.ts:140-152,269-285,313-326,405-417; src/routes/api/chat/voice-reply/+server.ts:55-60`
- ⚪ **[low]** bootstrapped/ensured guards are module-level booleans — safe for a single process, fragile if assumptions change
  - Table-creation is guarded by module-scoped one-shot flags: bootstrap.ts:21 `bootstrapped`, thread_state.ts:31 `ensuredPaths` Set keyed on the (constant) db path, dispatchJobs.ts:90 `_ensured`, chatActivity.ts:5 `_activityEnsured`. These assume one long-lived Node process and a never-changing memoryDbPath. They're correct under adapter-node today, but combined with the per-call connection pattern the 'ensure' is decoupled from the connection that uses it (a fresh connection trusts a flag set on a prior, now-closed connection). Latent footgun if the DB file is ever swapped at runtime or the process forks.
  - _evidence:_ `src/lib/server/bootstrap.ts:21; src/lib/server/thread_state.ts:31-33; src/lib/server/dispatchJobs.ts:90; src/lib/server/chatActivity.ts:5-7`

**Refactor opportunities:**
- (L) **Introduce a single db.ts owning the connection + all schema/migrations** — Replace the ~23 per-call `new Database()` openers with one shared module that opens the connection once (better-sqlite3 is synchronous and a single connection is the recommended pattern), sets WAL + busy_timeout in one place, and runs ALL CREATE TABLE/migrations on boot. This kills the pending_jobs/chat_activity schema duplication and the chat_activity index drift, removes the scattered ensure-flag pattern, and eliminates the SQLITE_BUSY-without-busy_timeout risk. bootstrap.ts becomes the migration manifest; per-feature modules just `import { db }`. _[src/lib/server/chat.ts:6-8; src/lib/server/bootstrap.ts:34-188; src/lib/server/dispatchJobs.ts:91-137; src/lib/server/chatActivity.ts:6-19]_
- (S) **Fix the chat_activity index drift immediately (point fix)** — Independent of the larger db.ts refactor: make bootstrap.ts:94 and chatActivity.ts:16 agree on the index definition (use the (trace_id, timestamp) composite everywhere, or rename one). Right now the index columns that exist at runtime depend on which module touched the cold DB first. _[src/lib/server/bootstrap.ts:94 vs src/lib/server/chatActivity.ts:16]_
- (M) **Decompose the legacy /api/chat POST and route it through prepareStream-style shared helpers** — The 5-branch god-handler duplicates repo-detection, conversation-slicing, and worker-prompt construction that already exist as shared functions for the sdk-stream path. ### FRONTEND — chat surface (src/routes/chat/+page.svelte, src/lib/components/*, src/lib/chat/*.svelte.ts)

**Wiring:** The chat surface is one route (`/chat/+page.svelte`, 1077 lines) acting as an orchestration shell over 11 controller factories in `src/lib/chat/*` and 8 presentational components in `src/lib/components/*`. The page owns the canonical reactive state that multiple writers touch (`messages`, `selectedRepo`, `currentTier`, `sending`, scroll state) and constructs each controller with an explicit deps "port" of getters/setters (e.g. `createThreadsController`, `createStreamingController`, `createVoiceController`, `createRealtimeVoiceController`, `createMessageActionsController`, `createComposerStateController`). Controllers own their own cohesive `$state` (threads list, talkback phase, stream state, per-message button-state Sets) and reach page state only through the port — reactive ownership stays where the template binds. Data flows in via `+page.server.ts` loader (`data.messages/threads/workspaces/appIdentity/companionDispatchEnabled`). Streaming uses `@ai-sdk/svelte` `Chat` against `/api/chat/sdk-stream`, mirrored into a placeholder bubble in `messages`; non-stream/dispatch/image sends hit `/api/chat`. Two distinct voice systems: legacy in-composer Talkback (`voice.svelte.ts`) and the full-screen realtime local-GPU pipeline (`realtime-voice.svelte.ts` + `VoiceMode.svelte`). `MessageFeed` renders the feed read-only and emits all actions via callbacks; `Composer` is fully prop/callback-driven with bindables for `textDraft`/`imageMode`/`isDragging`/`textareaEl`/`showModelOverrideModal`. A 3s `pollMessages` interval reconciles the feed against the DB. The page is wrapped by `+layout` chrome (ToastContainer, PwaUpdatePrompt — not in this audit scope).

**Strengths:**
- ✅ Controller decomposition is genuinely good and consistently applied: 11 factories each own a cohesive concern, expose $state only through getters (codebase convention noted in dispatchStream.svelte.ts:12), and reach page state via an explicit typed deps port. **Wiring:** Three live entry points feed one shared turn-service spine. ### DISPATCH & TASK-FIRST

**Wiring:** Every chat turn (text via sdk-stream, voice via voice-reply) mints a taskId in prepareStream/voice-reply and calls persistUserTurn → proposeTask(), which INSERTs a 'proposed' pending_jobs row (worker='sully') and journals task_proposed into chat_activity. classifyAndTouchThread journals classifier_ran. After the reply streams, the route calls maybeAutonomousDispatch (autonomous_dispatch.ts). On the CLI/Opus path the system prompt is suffixed with GATE_INSTRUCTION; the model appends a hidden <<<SULLY_GATE {...}>>> block which extractGateBlock strips from the visible reply and passes as gateBlock. maybeAutonomousDispatch runs ruleGate (@cc/@agy literal) + valueGate (regex objective-signal) and, on the CLI path, validateGate. If it fires, dispatchToWorker (companionDispatch.ts) runs the brakes (kill switch, 429 breaker, daily cap, fingerprint dedupe, token bucket), createJob() upserts the 'proposed' row to 'decided', writes a {prompt} JSON file, and HMAC-POSTs to the dispatch listener (markDispatched on 2xx, markFailed otherwise).
